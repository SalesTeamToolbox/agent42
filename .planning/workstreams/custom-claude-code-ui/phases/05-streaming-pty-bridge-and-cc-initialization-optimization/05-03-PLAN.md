<objective>
Add a pre-warmed CC session pool to eliminate the ~50s cold start delay. When a WebSocket connection opens with ?warm=true, the backend spawns a CC process with a lightweight sentinel message. CC initializes all MCP servers during this sentinel run. When the user sends their real first message, the backend uses --resume with the warm session_id, so CC starts near-instantly.

Purpose: CC takes ~50s to initialize 12+ MCP servers on first message. Pre-warming eliminates this wait by front-loading initialization to WS connection time.
Output: Modified dashboard/server.py with _cc_warm_pool dict, spawn/claim/cleanup logic
</objective>

**Step 1: Define the warm pool dict**

```python
    # PTY-03: Pre-warmed CC session pool (one warm process per user)
    import time as _time_cc
    _cc_warm_pool: dict = {}
    # Each entry: {
    #   "cc_session_id": str | None,  # From result event of sentinel run
    #   "ready": bool,                # True after sentinel run completes
    #   "spawned_at": float,          # time.monotonic()
    #   "last_used": float,           # time.monotonic() -- for idle timeout
    # }
```

**Step 2: Define the spawn warm function**

Create `_cc_spawn_warm(user: str, workspace_path)` as an async function inside create_app() closure. This function:

1. Finds claude binary via `_shutil.which("claude")`. Returns early if not found.
2. Builds warm_args: `[claude_bin, "-p", ".", "--output-format", "stream-json", "--verbose"]` -- the sentinel message is `"."` (a single period, minimal token cost).
3. Spawns the CC process using PTY if available (same platform detection as cc_chat_ws), else falls back to PIPE.
4. Reads the output stream looking for a `"result"` event. Extracts `session_id` from it.
5. On success: stores `{"cc_session_id": warm_session_id, "ready": True, "spawned_at": now, "last_used": now}` in `_cc_warm_pool[user]`.
6. On failure: logs warning, does not store anything.

Windows PTY path: Use `PtyProcess.spawn(warm_args, cwd=..., dimensions=(24, 220))`, then `readline` loop via `run_in_executor` until `result` event.
Unix PTY path: Use `pty.openpty()` + `Popen`, then `select` + `os.read` line accumulation loop until `result` event. Close master_fd after.
PIPE fallback: Use `create_subprocess_exec` with PIPE stdout, `async for` loop until `result` event.

All paths strip ANSI codes via `_ANSI_ESCAPE.sub("", line)` and skip non-JSON lines.

**Step 3: Define the idle cleanup background task**

```python
    async def _cc_prewarm_idle_task():
        """Background task: remove warm pool entries older than 5 minutes (300s)."""
        while True:
            await _asyncio.sleep(60)  # Check every 60 seconds
            now = _time_cc.monotonic()
            expired = [
                user for user, entry in _cc_warm_pool.items()
                if now - entry["last_used"] > 300  # 5 * 60 = 300 seconds
            ]
            for user in expired:
                del _cc_warm_pool[user]
                logger.info(f"CC warm session expired for {user}")
```

**Step 4: Start the cleanup task at app startup**

Use FastAPI startup event:
```python
    @app.on_event("startup")
    async def _start_warm_pool_cleanup():
        _asyncio.create_task(_cc_prewarm_idle_task())
```

**Point 2: First user message -- check warm pool before spawning**

Inside the per-message `while True` loop, AFTER the claude_bin check but BEFORE the PTY spawn block, add:

```python
                # PTY-03: Check warm pool for pre-initialized session
                warm_entry = _cc_warm_pool.pop(_cc_user, None)
                if warm_entry and warm_entry.get("ready") and warm_entry.get("cc_session_id"):
                    if not session_state.get("cc_session_id"):
                        session_state["cc_session_id"] = warm_entry["cc_session_id"]
                        logger.info(f"CC using warm session {warm_entry['cc_session_id']} for {_cc_user}")
                        await websocket.send_json({
                            "type": "status",
                            "data": {"message": "Using pre-warmed session"}
                        })
```

This works because the existing args builder already checks `session_state.get("cc_session_id")` and appends `--resume` if present. So the warm session_id flows through the existing code path with zero additional changes.

**Key implementation notes:**
- `_cc_warm_pool.pop(_cc_user, None)` atomically removes the entry, preventing double-claim from concurrent tabs
- The warm check only sets session_state when `cc_session_id` is None (first message only). Subsequent turns already have a cc_session_id from the result event.
- If the warm entry is not ready (sentinel still running), it is popped and discarded -- the message proceeds with a fresh spawn. This is safe because the warm spawn task will complete harmlessly in the background and the entry it tries to store will be for a user who already has a session.
- The warm process itself (sentinel run) is already dead by the time the user sends a message. There is no live process to clean up.
- Only one warm process per user (locked decision) -- pool key is the username string
